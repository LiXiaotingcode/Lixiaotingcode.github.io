Docker 学习干货整理：从镜像到部署的完整入门笔记

发表于2026-05-10|更新于2026-05-10|Docker学习
|浏览量:

Docker 的价值不只是“能跑起来一个环境”，而是把应用、依赖、运行方式一起打包，让开发、测试、部署尽量保持一致。

这篇文章按“从概念到实战”的顺序整理，适合初学者作为第一篇完整笔记。

1. Docker 到底解决了什么问题

在没有 Docker 之前，开发环境和生产环境经常不一致：

  • 你电脑能跑，服务器跑不起来
  • 依赖版本不同
  • 配置文件位置不同
  • 手动部署步骤太多，容易漏

Docker 的核心价值是：

  • 环境一致
  • 部署更快
  • 隔离更清晰
  • 迁移更方便

2. 先分清镜像、容器、仓库

这是学习 Docker 最容易混淆的三个概念。

  • 镜像 image：应用运行的模板
  • 容器 container：镜像启动后的运行实例
  • 仓库 registry：存放镜像的地方，比如 Docker Hub

可以这样理解：

  • 镜像像安装包
  • 容器像安装包运行后的程序
  • 仓库像软件下载站

3. 安装后先会这组基础命令

docker --version
docker info
docker images
docker ps
docker ps -a
docker pull nginx
docker run nginx
docker stop <container_id>
docker rm <container_id>
docker rmi <image_id>

高频动作其实就这几类：

  • 拉镜像
  • 启动容器
  • 看日志
  • 进容器
  • 停止和删除

4. 最常用的启动方式

docker run -d --name my-nginx -p 8080:80 nginx

这条命令可以拆解为：

  • -d：后台运行
  • --name my-nginx：给容器命名
  • -p 8080:80：把宿主机 8080 映射到容器 80 端口
  • nginx：使用的镜像名

查看是否成功：

docker ps
curl http://127.0.0.1:8080

5. 日志和进入容器是排障基本功

docker logs my-nginx
docker logs -f my-nginx
docker exec -it my-nginx /bin/bash
docker exec -it my-nginx /bin/sh

如果容器启动后立刻退出，先不要猜，先看日志。

6. 数据卷为什么重要

容器删除后，容器内部数据默认可能一起丢失，所以要学会用数据卷。

docker run -d \
  --name my-nginx \
  -p 8080:80 \
  -v /data/nginx/html:/usr/share/nginx/html \
  nginx

这样宿主机目录和容器目录就挂载起来了：

  • 改宿主机文件，容器里同步可见
  • 删除容器，宿主机数据还在

常见场景：

  • 挂载网站静态文件
  • 挂载数据库数据目录
  • 挂载配置文件
  • 挂载日志目录

7. Docker 网络要理解端口映射

很多新手会混淆“容器内部端口”和“宿主机暴露端口”。

docker run -d --name web -p 3000:80 nginx

含义是：

  • 容器内部服务监听 80
  • 宿主机通过 3000 对外提供访问

所以你访问的是：

http://127.0.0.1:3000

查看端口映射：

docker port web

查看网络：

docker network ls
docker network inspect bridge

8. 镜像是怎么做出来的

Docker 不只是拉别人的镜像，你也会自己构建镜像。

一个最简单的 Dockerfile：

FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["npm", "start"]

构建镜像：

docker build -t my-app .

启动容器：

docker run -d --name my-app -p 3000:3000 my-app

9. 写 Dockerfile 时要注意什么

几个常见原则：

  • 基础镜像尽量小，比如 alpine
  • 先复制依赖文件，再安装依赖，利用构建缓存
  • 不要把无关文件都拷进去
  • 明确工作目录
  • 明确启动命令

建议加一个 .dockerignore，避免把 node_modules、日志、构建产物带进镜像。

10. Docker Compose 更适合多服务场景

当项目不止一个服务时，比如：

  • 前端
  • 后端
  • MySQL
  • Redis

这时候单独写很多 docker run 会很乱，更适合用 Compose。

示例 compose.yaml：

services:
  web:
    image: nginx:latest
    ports:
      - "8080:80"
  redis:
    image: redis:latest

启动：

docker compose up -d

停止：

docker compose down

查看状态：

docker compose ps

11. 常用排障命令要形成肌肉记忆

docker ps -a
docker logs <container>
docker inspect <container>
docker exec -it <container> /bin/sh
docker stats
docker network ls
docker volume ls

重点看这几类信息：

  • 容器是否真的在运行
  • 启动日志有没有报错
  • 端口是否映射正确
  • 配置文件是否挂载成功
  • 环境变量是否注入成功

12. 常见问题排查思路

12.1 容器启动就退出

先查：

docker logs <container>

可能原因：

  • 启动命令写错
  • 依赖没装全
  • 应用本身异常退出
  • 配置文件缺失

12.2 端口访问不到

先查：

docker ps
docker port <container>
curl http://127.0.0.1:映射端口

再看：

  • 服务是不是监听在容器正确端口
  • 是否映射到了宿主机
  • 防火墙是否拦截

12.3 数据没保存

通常是没挂载卷，或者挂载路径写错。

12.4 镜像太大

通常是：

  • 基础镜像太重
  • 把无关文件复制进去了
  • 没有清理缓存

13. 学 Docker 的推荐路线

建议按这个顺序学：

  1. 镜像、容器、仓库概念
  2. 常用 docker 命令
  3. 端口映射和日志查看
  4. 数据卷与持久化
  5. Dockerfile
  6. Compose
  7. 镜像优化
  8. 线上部署和排障

14. 常用命令速查表

docker images
docker ps
docker ps -a
docker pull nginx
docker run -d --name web -p 8080:80 nginx
docker logs -f web
docker exec -it web /bin/sh
docker stop web
docker rm web
docker rmi nginx
docker build -t my-app .
docker compose up -d
docker compose down

15. 总结

Docker 的学习重点，不是背所有参数，而是建立稳定的心智模型：

  • 镜像是模板
  • 容器是运行实例
  • 端口映射决定访问方式
  • 数据卷决定数据是否持久化
  • 日志和 inspect 是排障入口

一旦这些概念清楚了，后面学 Compose、Kubernetes、CI/CD 会顺很多。
